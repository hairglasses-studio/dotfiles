# Decision Journal

Capture architectural decisions at the moment they happen so the reasoning survives context compaction, session handoff, and the next person touching the code. Append-only log under `docs/decisions.md`, one short ADR-lite entry per decision.

## When to use

Capture when:

- A design choice got made in conversation — "going with X", "chose X over Y because Z"
- User says "log this decision", "decision journal", "capture this ADR", "record that we chose …"
- At the end of a planning thread that settled a non-trivial call
- When refactoring rationale is non-obvious from the diff alone ("we're splitting this because…")

Skip when:

- The choice is a local implementation detail with no cross-cutting impact (variable name, loop style)
- The decision is already captured in an existing entry — verify first via grep before writing
- The user is rubber-ducking and hasn't actually settled on anything

## Invocation

`$ARGUMENTS` may be:

- Empty — capture the most recent decision from the conversation, using your own summary
- `--title "<short title>"` — force a specific entry title
- `--export` — emit the current `docs/decisions.md` as a markdown table (date, title, decision) and do not write anything
- `--path <dir>` — target a directory other than `docs/` (useful for repos without `docs/`)

## Workflow

### Phase 1 — Detect the decision

If the user invoked the skill with an explicit statement, use it. Otherwise, scan the last few turns for a sentence of the form:

- "we decided to X"
- "going with X because Y"
- "chose X over Y"
- "let's X"

If nothing clear is present, ask the user for the one-line decision before writing. Never invent a decision.

### Phase 2 — Check for duplicates

```bash
grep -n "^## " docs/decisions.md 2>/dev/null | tail -20
```

If a heading with the same title and current date already exists, report the existing entry's line number and skip writing. Append only when the decision is new.

### Phase 3 — Write the entry

Target path: `docs/decisions.md` (create if missing, with a one-line header `# Decision Log`). Append in reverse-chronological order — newest entry at the top, immediately after the header.

Entry template:

```markdown
## YYYY-MM-DD — <Short imperative title>

**Context**: <One-to-two sentence summary of the situation that forced a choice. What constraints or tradeoffs were in play.>

**Decision**: <What we chose. One sentence.>

**Rationale**: <Why. One or two sentences. Include concrete evidence if available — benchmark, incident, stakeholder ask.>

**Alternatives considered**: <Bulleted list of alternatives actually weighed, with a one-line note on why each was rejected. Omit this section if only one path was ever on the table.>

**Consequences**: <What this commits us to, what it blocks, what becomes easier. Omit if trivial.>
```

Keep each entry under 15 lines. If more detail is needed, link to a longer doc — do not inline the whole thing.

### Phase 4 — Confirm

After writing, print:

- Path to the updated `docs/decisions.md`
- Line numbers of the new entry
- The title of the new entry, quoted

Do not stage, commit, or push — the author handles git.

## Export mode

With `--export`:

```bash
awk '/^## [0-9]{4}-[0-9]{2}-[0-9]{2}/ { print }' docs/decisions.md
```

Convert each `## YYYY-MM-DD — Title` heading into a markdown table row:

```markdown
| Date | Title | Decision (one line) |
|------|-------|---------------------|
| 2026-04-21 | Flycast core | Switched NGC/Dreamcast to Flycast (bsnes-hd keeps SNES) |
```

Emit the table to stdout only. Do not overwrite the source log.

## Anti-patterns

- Don't write vague decisions ("we improved the code") — if the decision isn't concrete and reversible, it's not a decision worth logging
- Don't duplicate context that's obvious from the current codebase — link to `git blame`/`git log` evidence instead of re-stating it
- Don't bulk-import past decisions from chat history without user confirmation — only append what the user has just settled
- Don't auto-tag entries with issues/PRs Claude can't verify exist
- Don't spread entries across multiple files — one append-only log per repo
- Don't reformat or reorder existing entries; always append to the top of the list below the header
